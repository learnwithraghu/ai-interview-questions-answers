Question 09

Debugging RAG: Retrieval or Generation?

You are building a document Q&A bot. Users complain that the bot says "I don't know" even when the answer is clearly in the document. How do you troubleshoot whether the failure is in the retrieval step or the generation step?

Isolate before you fix anything.

Visual Explanation

Three-step isolation process

1
Log retrieved chunks for the failing queryRead the raw chunks the vector DB returned — are they even relevant?
2
Manually inject the correct chunk, bypass retrieval entirelyCopy the right chunk directly into the prompt and send it to the LLM
3
Observe: does the model answer correctly with the injected chunk?Yes → retrieval is broken. No → generation is broken.
Step 1 is critical: logging chunks is the single most important debugging step in any RAG pipeline. If chunks are wrong: retrieval failed — fix chunking strategy, embedding model, or similarity threshold. Step 2 bypasses retrieval: this isolates generation in a clean, controlled test — no vector DB involved. If LLM still fails: check if the system prompt has an overly strict refusal instruction ("only answer if 100% certain"). Truncation trap: context being cut off silently is a common generation failure — check token limits. Root cause first: never touch retrieval and generation settings simultaneously — that destroys reproducibility.

Visual Explanation

Two paths, two fix sets

Retrieval Broken
Wrong chunks in logs →
✦ Improve chunking (semantic > fixed)
✦ Lower similarity threshold
✦ Switch embedding model
✦ Add metadata filtering or re-rank
Generation Broken
Right chunks, model still fails →
✦ Relax system prompt refusal rules
✦ Fix context formatting
✦ Increase context window
✦ Use model better at instruction-following
Legal firm example: bot said "I cannot find that information" — logs showed unrelated chunks — a retrieval fix. Semantic chunking: splits on meaning boundaries, not fixed character counts — preserves context across chunk edges. Threshold tuning: lowering similarity threshold trades precision for recall — more chunks, fewer misses. Formatting matters: markdown tables or raw SQL in chunks confuse some models — clean text extraction helps. Prompt audit: "only answer if you are 100% certain" is a common source of false "I don't know" responses. Track both: measure retrieval precision and generation faithfulness as separate production metrics.

Answer

Say it

"I treat RAG as two separate systems. First, I'd log all retrieved chunks for the failing queries to check if the right content is actually being fetched. If the right chunks are there but the model still says 'I don't know', I know it's a generation problem. If the wrong chunks are being retrieved, that's a retrieval problem. The key is isolating which side is failing before touching anything."
