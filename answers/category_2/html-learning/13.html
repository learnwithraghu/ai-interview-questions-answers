Question 13

LLM-as-a-Judge Evaluation

You are evaluating two different LLMs for a summarization task. One is cheaper but potentially less accurate. How do you set up an automated evaluation pipeline to quantify the trade-off at scale?

Quantify the trade-off — don't eyeball it.

Visual Explanation

The LLM-as-a-judge pipeline
Golden Dataset
100–500 representative inputs + reference outputs
Run Both Models
Model A (expensive) + Model B (cheap) on same inputs
Judge LLM Scores
GPT-4o rates: accuracy, conciseness, relevance (1–5 JSON)
Decision Matrix
Score vs cost per 1000 calls → accept/reject threshold
Why not ROUGE/BLEU: word-overlap metrics say nothing about factual correctness or coherence — they miss the point. Golden dataset quality: 100–500 curated inputs reflecting real production distribution — garbage in, garbage out. Judge prompt: "Rate this summary 1-5 on Factual Accuracy, Conciseness, Relevance. Return JSON." — structured and repeatable. Use a different provider for the judge: judge models favour outputs similar to themselves — cross-provider judging reduces bias. Randomise order: present outputs to the judge in random order to avoid position bias in scoring. Cost metric: measure cost-per-1000-API-calls (input + output tokens), not just listed token price.

Acceptance threshold: define it BEFORE looking at results — "within 10% on factual accuracy" prevents unconscious bias. Factual accuracy wins: it's usually the most important dimension — conciseness matters less if facts are wrong. Media example: Gemini Flash 4.1 vs GPT-4o 4.4 — within 10% — deployed Flash for 80% cost savings on non-critical summaries. Tiered strategy: cheap model for bulk tasks, expensive model as fallback for high-stakes or edge cases. Scale signal: run the evaluation monthly as models update — today's cheaper model might improve and overtake the expensive one. Judge bias caveat: LLM judges prefer longer responses — compensate with explicit rubrics and cross-provider judging.

Answer

Say it

"I'd build a benchmark dataset of representative inputs with human-written reference summaries. Then I'd run both models across the benchmark and use a powerful LLM like GPT-4o as the judge — scoring each output on factual accuracy, conciseness, and relevance using a structured rubric. That gives me quantitative scores I can compare side by side with cost. If the cheaper model scores within an acceptable threshold on the most critical metric, the cost savings justify using it — with the expensive model as a fallback for edge cases."
