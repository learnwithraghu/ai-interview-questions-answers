Question 08

RAG Fails Independently

In a RAG system, why should retrieval and generation be treated as two separate systems that can fail independently?

RAG is two systems glued together, not one magic feature.

Retrieval fail: right information never reaches the model — the LLM cannot answer from evidence it never saw. Generation fail: right chunks arrived, but the model ignored them or misread the context. Key insight: if retrieval is broken, prompt changes are a distraction — fix chunking, embeddings, or thresholds. Key insight: if generation is broken, changing the vector DB is a distraction — fix the prompt or context format. Log the boundary: record the exact query, retrieved chunks, scores, and the final prompt sent to the model. Outcome: once you can see the boundary, you pinpoint which side failed instead of guessing randomly.

Visual Explanation

Fix depends on the failure side

Retrieval Broken

  • Chunks too small, answer split
  • Embedding misses domain vocab
  • Similarity threshold too strict
  • Fix: re-chunk, re-embed, re-rank

Generation Broken

  • Prompt too restrictive
  • Context truncated or poorly formatted
  • Model ignores retrieved evidence
  • Fix: adjust prompt, format chunks better
Diagnosis first: log chunks for every failing query and read them before touching any configuration. Retrieval test: manually inspect if the correct chunk exists in the top results — if not, retrieval failed. Generation test: inject the correct chunk directly into a prompt and test the LLM in isolation. Support bot example: wrong answer on refund eligibility could be missing chunks (retrieval) or ignored policy clause (generation). No random tuning: without this separation, teams waste weeks tweaking the wrong system. Monitoring: track retrieval hit-rate and generation quality as separate metrics in production.

Answer

Say it

"I treat RAG as two systems: retrieval finds the evidence, and generation turns that evidence into an answer. They can fail independently. If the right chunks are missing, that's a retrieval problem. If the right chunks are present but the model still answers badly, that's a generation problem. Separating those two failure modes helps me debug systematically instead of randomly changing prompts or vector settings."
