Question 10

Natural Language to SQL

Your application uses an LLM to generate SQL queries from natural language. How do you handle database schema changes and prevent the LLM from hallucinating column names?

Ground it in the real schema, validate before you execute.

Core problem: LLMs invent column names with confidence when they have no schema — hallucination by default. Schema injection: dynamically fetch real table names, column names, types, and FK relationships at query time. Large DB: embed table descriptions and retrieve only relevant tables first — avoid hitting token limits. Few-shot examples: 2–3 NL→SQL pairs in the system prompt teach the model your JOIN and WHERE patterns. Validation gate: parse the generated SQL, extract column references, cross-check against the live schema before running anything. Reject and retry: if validation fails, feed the error back with the correct schema — never execute unvalidated SQL.

Visual Explanation

Handling schema changes
Schema Sync
Hook database migration tool (Alembic, Flyway) to regenerate your stored schema definition on every migration run.
Never Execute Blind
Parse every generated SQL query and verify every table and column reference against the live DB schema before running.
Retry With Error
If a column doesn't exist, return the error + the correct schema snippet to the LLM and ask it to regenerate.
Selective Injection
For large databases, embed table descriptions and retrieve only the relevant tables — keeps prompt focused.
Migration hook: the schema definition auto-updates when any migration runs — stale schema = broken queries. Why validate: even perfect schema injection can't catch every edge case — validation is the final safety gate. E-commerce example: analysts query a sales DB in plain English — CI/CD hook keeps the schema prompt always current. Selective injection: injecting 200 tables into a prompt wastes tokens and confuses the model — retrieve only what's needed. Few-shot matters: without examples, models often produce syntactically valid but semantically wrong JOIN logic. Prod rule: treat LLM-generated SQL like user input — always validate, never trust, before hitting production data.

Answer

Say it

"The main strategy is schema injection — I dynamically pull the current table schemas and inject them into the system prompt so the model only knows columns that actually exist. To handle schema changes, I automate the schema sync so it updates whenever a migration deploys. Before executing any generated SQL, I validate the column names against the live DB — if a column doesn't exist, I reject the query and retry with a corrected prompt."
