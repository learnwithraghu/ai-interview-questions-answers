Question 12

Advanced RAG Retrieval Strategies

Your RAG application is returning irrelevant context for complex, multi-part user queries. What advanced retrieval strategies — query expansion, HyDE, cross-encoder re-ranking — would you implement to fix this?

One vector can't represent two intents at once.

Visual Explanation

Four strategies to fix complex queries

🔀
Query DecompositionBreak complex question into simpler sub-queries → retrieve separately → merge results
💭
HyDEAsk LLM to write a hypothetical answer → embed that answer → use it for retrieval
⚖️
Cross-Encoder Re-rankingFast bi-encoder gets top-50 candidates → cross-encoder scores query+chunk pairs → keep top-5
🔀
Hybrid Search (BM25 + Dense)Run keyword search and vector search in parallel → merge with Reciprocal Rank Fusion
Root cause: a single query vector cannot optimally represent multiple intents — complex questions always lose context. Decomposition first: this is the most impactful and cheapest fix — break the question, retrieve independently. HyDE trick: an "answer-shaped" embedding is semantically closer to document chunks than a "question-shaped" one. Re-ranking is two-stage: fast bi-encoder for recall, accurate cross-encoder for precision — never run cross-encoder on the full corpus. Hybrid wins on exact terms: product codes, legal terms, and proper names need keyword search — pure vector search misses them. RRF merging: Reciprocal Rank Fusion combines rankings from both keyword and vector lists into one unified ranking.

Why two stages: cross-encoders are too slow for full corpus search but highly accurate — bi-encoders shortlist first. Bi-encoder: embeds query and document separately, compares with cosine similarity — scales to millions of chunks. Cross-encoder: takes query and candidate chunk as a single input — sees their full interaction, scores relevance precisely. Finance example: complex Q3 risk + 2025 guidance query is decomposed into two sub-queries, each retrieves top-50, cross-encoder selects top-10. When to add: if simple vector search accuracy is above 70%, add re-ranking; below that, fix chunking first. Cost tradeoff: each strategy adds latency — start with decomposition, add re-ranking when precision matters most.

Answer

Say it

"A single vector query breaks down for complex questions because one vector can't capture multiple intents. My go-to fix is query decomposition — break the complex question into simpler sub-queries and retrieve independently for each. I'd also consider HyDE, where I ask the LLM to generate a hypothetical answer and use that for retrieval since it's semantically closer to the actual documents. On top of that, I'd add a cross-encoder re-ranker as a second pass to filter down merged results to the most relevant chunks."
