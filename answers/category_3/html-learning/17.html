AI Engineer Interview — Q17

Securing AI Agents

"A malicious prompt injection attack successfully tricks your AI agent into executing an unauthorised API call using its connected tools. How do you secure the agent's function-calling capabilities and implement a human-in-the-loop safeguard?"

💡 Giving an LLM real tools changes the security model completely.

AI Engineer Interview — Q17 — Defence in Depth

Four security layers

1
Input scanning — Classifier detects instruction-override patterns before the agent sees the message.
2
Least privilege — Agent only has the tools it genuinely needs; narrow scope = smaller blast radius.
3
Argument validation — Validate every tool argument against a strict schema before execution.
4
HITL gate — Irreversible actions pause and require human approval before they run.
Indirect injection: Malicious payload embedded in a support ticket, document, or tool output — not direct user input. Blast radius: If injection gets through, the attacker can only do what the tools allow. Keep tools minimal. Schema validation: If issue_refund expects $0–$500, reject any call where amount exceeds that. Defence in depth: No single layer stops everything — all four layers must be present.

Human-in-the-loop gate

User request

↓  HITL gate checks reversibility  ↓
Gate decision
Reversible / Read-only → Auto-execute ✓
Web search • DB read • Weather lookup
Irreversible → Human confirms ⚠
Send email • Make payment • Delete record
Reversible = safe: Worst case is a wasted API call — auto-run is fine. Irreversible = show human: Exactly what will happen, and require a click before it runs. Audit log: Record every tool call, arguments, and outcome — especially for emails and payments. Max iteration guard: Set a step limit (e.g. 10) — agents can loop indefinitely if a tool keeps failing.

AI Engineer Interview — Q17 — Say It

Say it

"My approach is defence-in-depth. At the input level, I'd run every user message through an injection detection classifier. At the tool level, I'd apply least privilege — only expose tools the agent actually needs, validate all arguments against a strict schema, and separate tools into safe and destructive categories. For anything irreversible, I'd implement a hard human-in-the-loop gate that pauses the agent and requires explicit human approval before execution."

Input scan → Least privilege → Schema validation → HITL gate for irreversible actions.
