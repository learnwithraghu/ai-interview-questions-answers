Hard limit: GPT-4o is 128K tokens, Claude is 200K — but the limit includes EVERYTHING you send. Lost in the middle: LLMs underweight information in the middle of a very long context. History grows: Every turn appends more messages — a busy session can hit the limit in hours. Fix timing: Check token count before every call and compress at 80% of the limit, not after hitting it.

AI Engineer Interview — Q15 — Compression Strategies

Three compression strategies

1
Sliding Window — Keep only the last N messages. Simple, but drops early context.
2
Conversation Summarization — Compress old messages with a cheap LLM (GPT-4o-mini). Preserves meaning, cuts tokens 10x.
3
External Memory Store — Extract key facts to a vector DB. Retrieve only what's relevant per query.
Sliding window flaw: If the user's name was in message 1, a sliding window loses it forever. Summarization sweet spot: Keep last 10 messages verbatim + a 2–3 sentence summary of everything older. Token counting: Use tiktoken (OpenAI) or count_tokens (Anthropic) before every API call — not after. Production result: A hybrid sliding+summary approach cut timeouts from 30% to under 0.5%.

AI Engineer Interview — Q15 — Say It

Say it

"First I'd add proactive token counting before every API call — if we're approaching the limit, trigger compression before the timeout happens. My preferred strategy is conversation summarisation: once the history gets too long, I use a fast, cheap model to compress older messages into a short summary and replace them with that. I'd combine that with a sliding window that always preserves the last 10 messages verbatim. For power users, I'd also extract key entities into an external memory store."

Count tokens proactively → summarise old messages → sliding window the rest → external memory for key facts.
