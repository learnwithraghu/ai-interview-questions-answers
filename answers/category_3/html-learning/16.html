API Rate Limit Fallbacks

"During a product launch, your primary LLM API provider hits a severe rate limit (HTTP 429), causing complete service failure. How do you implement robust fallback mechanisms and load balancing across different models?"

💡 Treat LLM providers like any other third-party dependency — design for failure.

LiteLLM / Portkey: Proxy layers between your app and multiple providers — one consistent API for all. Fallback chain: Define priority order. OpenAI → Anthropic → Gemini. Proxy shifts automatically on 429. Your app never changes: Add or swap providers in config, not in application logic. Circuit breaker: If a provider fails 5 times in 30s, stop routing to it for a 60s cooldown.

Exponential backoff: 1s → 2s → 4s delay between retries. Never hammer a struggling provider. Max retries: Set to 3. After that, fail over to the next provider — don't keep waiting. Request queue: Put a queue in front of the LLM layer to absorb traffic spikes instead of dropping requests. Recovery: After cooldown, send a test request. If it succeeds, close the circuit and resume routing.

AI Engineer Interview — Q16 — Say It

Say it

"I'd architect the LLM layer with a proxy like LiteLLM so I can route to multiple providers without changing application code. I'd configure a fallback chain — OpenAI first, then Anthropic, then Gemini — so if one provider hits rate limits, traffic automatically shifts. On top of that, I'd implement retry logic with exponential backoff so we don't hammer a struggling provider, and a circuit breaker to stop routing to one that's consistently failing."

Proxy layer → Fallback chain → Exponential backoff → Circuit breaker → Request queue.
