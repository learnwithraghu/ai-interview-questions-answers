# Question 20: Open-Source Versus Cloud Agents

**Question:** Open-source agent frameworks (e.g., CrewAI, Pydantic AI) and cloud-native agent platforms (e.g., OpenAI's Agents SDK, AWS Bedrock Agents) take fundamentally different approaches to building AI agents. Which one are you using in your own projects, and what trade-offs drove that choice?

**Answer:**

### 1. Question Explanation
This question isn't really about naming frameworks — it's about whether you've actually shipped an agent and had to live with the consequences of a tooling choice. Interviewers ask this to see if you understand that "agent framework" isn't one category but a spectrum: fully open-source, self-hosted code on one end, and fully managed, cloud-run infrastructure on the other. A weak answer lists framework names like a changelog. A strong answer picks a side, explains the specific constraint that pushed you there — cost, latency, data residency, how fast you needed to ship — and shows you know what you gave up by not choosing the other path.

### 2. Concept Explanation
Start with what an "agent framework" actually has to do under the hood, regardless of which one you pick: it needs to hold a loop (call the model, parse what it wants to do, run a tool, feed the result back), manage state across that loop, and give you some way to see what happened when it breaks. **Open-source frameworks** like CrewAI and Pydantic AI hand you that loop as code you own. CrewAI gives you a "crew" abstraction — multiple role-based agents (researcher, writer, reviewer) coordinating on a shared task — and you control every prompt, every retry, every tool definition, because it's all just Python running on your own infrastructure. Pydantic AI takes a narrower, type-safe angle: it wraps your model calls so that tool inputs and outputs are validated against Pydantic schemas before your code ever sees them, which catches a whole class of "the LLM returned malformed JSON" bugs at the framework level instead of in your business logic. The shared trade-off across open-source options is that **you own the ops**: you pick the host, you build the observability, you patch the framework when it has a bug, and you pay only for the compute you use — which can be cheap at small scale and a real engineering burden at large scale.

**Cloud-native platforms** like OpenAI's Agents SDK and AWS Bedrock Agents flip that trade-off. They run the orchestration loop on the provider's infrastructure, so tool execution, conversation state, and tracing come built-in — you define the tools and the system prompt, and the platform handles retries, session memory, and step-by-step execution logs without you standing up any of that yourself. Bedrock Agents goes further by wiring directly into the AWS ecosystem — Lambda for tool execution, Knowledge Bases for retrieval, IAM for permissions — so if your company already lives in AWS, you inherit security and compliance controls you'd otherwise have to build by hand. The cost of that convenience is **control and portability**: you're bound to that provider's model lineup, their pricing model, their rate limits, and migrating off later means rewriting your orchestration layer, not just swapping an API key.

Here's the part that trips people up: this isn't an either-or for your whole company, it's a per-project decision. A regulated enterprise with an existing AWS footprint and a compliance team breathing down its neck has a very different optimal answer than a two-person startup trying to validate an idea in a weekend. The interviewer isn't grading you on which side you picked — they're grading you on whether you can name the actual constraint (latency budget, data residency, team size, time-to-ship) that makes one side correct for your situation.

### 3. Real-World Example
A fintech startup prototyping a research agent for internal use picks OpenAI's Agents SDK because they need something running in a day, don't have the headcount to build observability tooling, and aren't yet handling regulated customer data — the managed tracing and built-in tool-calling loop let two engineers ship in an afternoon. Six months later, once they're processing real customer financial data and need to guarantee it never leaves their VPC, they migrate the core agent loop to a self-hosted CrewAI setup running on their own AWS account, trading the convenience of the managed SDK for the data residency and cost control they now need at scale. Both choices were correct — for the constraint that existed at that point in time.

### 4. This is how I would answer this
"In my own projects I default to open-source — usually CrewAI when I need multiple coordinating agents, or Pydantic AI when I need strict, validated tool outputs — because I want the orchestration logic in my own codebase, not behind someone else's API. That matters most when I need to swap models later or keep data in my own infrastructure for compliance reasons. That said, when I'm prototyping something fast and don't have an observability stack ready yet, I'll reach for a managed platform like OpenAI's Agents SDK or Bedrock Agents — the built-in tracing and tool execution save real time, and I'm fine paying that flexibility cost for a short-lived prototype. The decision really comes down to whether I value control and portability, or speed and managed infrastructure, more for that specific project."
