Question 24  ·  AI Engineer Interview Bootcamp

OpenAI Function Calling

Walk me through exactly how you would use OpenAI's Function Calling API to allow an LLM to interact with an external REST API (like checking the weather).

The model decides what to call. Your code actually calls it.

Question 24  ·  The two-phase architecture

Model decides, code executes

Model side — decides
User: "Will it rain in Seattle tomorrow?"
→
LLM with function schema
→
Returns: get_weather({location:'Seattle', date:'2025-06-15'})
↓   your code takes over   ↓
Your code — executes
Validate args
→
Call weather REST API
→
Return result to model → user reply
Schema definition: function name + description + JSON Schema parameters. The model uses the description to decide when to call it. Model output: a JSON object — not executable code. Your backend reads it and decides whether to run the function. API keys stay secret: the model never touches the weather API. Only your backend does. Optional final turn: send the API result back to the model to generate a human-friendly reply like "Yes, 70% chance of rain."

Question 24  ·  Strict mode + parallel calls

Two power features

strict: true
  • Forces model to only output args that exactly match the schema
  • Eliminates argument hallucination
  • Required fields cannot be skipped
Parallel tool calls
  • Model can return multiple function calls at once
  • e.g. get_weather + get_calendar in one response
  • Cuts round-trips for complex tasks
Strict mode: schema compiled into constrained decoding at inference time — model cannot produce invalid JSON. Anthropic tool use: same mechanics, slightly different API structure, supports parallel calls natively. Gemini function calling: adds mode control — ANY (must call a function), NONE (must not call), AUTO (model decides). Tool descriptions matter: vague names cause misfires. "query_database" beats "do_thing" every time.

Question 24  ·  Say it

Say it

"I'd define the REST integration as a typed function schema — name, description, and parameter types — and register it with the function calling API. I'd send that definition to OpenAI alongside the user query, and let the model choose whether to invoke the function. My backend validates the generated arguments, calls the external weather API, and then either returns the raw data or lets the model summarise it into a friendly reply for the user."

Tip: emphasise that the model never calls the external API directly — this shows you understand the security boundary.
