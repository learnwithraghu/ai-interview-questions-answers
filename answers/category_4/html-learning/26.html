Question 26  ·  AI Engineer Interview Bootcamp

Caching LLM Responses

How do you handle caching LLM responses to reduce API costs and latency? Explain the architectural difference between exact-match caching and semantic caching.

Start simple — exact match. Layer in semantic caching only when hit rates justify the complexity.

Question 26  ·  Two caching strategies

Exact match vs semantic

Exact-Match Cache
  • Hash the prompt → check key-value store → return hit
  • Simple + safe
  • Only helps identical requests
Semantic Cache
  • Embed the prompt → compare to cached query embeddings → return if similar enough
  • Higher hit rate
  • Risk of returning wrong answer
Exact-match: hash the full prompt as the cache key. Hash hit = return cached response instantly. Zero risk. Semantic cache: embed the new query, run similarity search against prior query embeddings. Hit if cosine > threshold. Threshold risk: set threshold too low and you return a cached answer for a semantically different question. Safe domains for semantic caching: FAQs, recommendations, canned summaries. Avoid for factual, time-sensitive queries.

Question 26  ·  Layered caching strategy

Four layers, in order

1
Server-side prompt caching (Anthropic/OpenAI)Free, automatic, 50–90% cost reduction on repeated prefixes.
2
Exact-match application cache (Redis/Memcached)Identical requests return instantly, zero LLM cost.
3
Semantic cache (Portkey/GPTCache)Paraphrased queries hit the cache, 20–40% additional savings.
4
Fresh LLM callAll caches miss, full token cost paid.
Anthropic prompt caching: mark system prompt with cache_control breakpoint. 90% discount on cached prefix tokens. OpenAI auto-caching: first 1024 tokens auto-cached at 50% discount for 1 hour. No code change needed. Portkey semantic cache: managed semantic cache as a gateway feature. No embedding infra to maintain yourself. Layer 1 first: apply provider-level prompt caching before any application code — it's free savings.

Question 26  ·  Say it

Say it

"I'd start with exact-match caching for identical requests because it's simple and reliable — hash the prompt, store the response, return on hit. For broader query reuse, I'd add semantic caching with embeddings so paraphrased questions can hit the cache too, but only after validating that the reused response is still appropriate. I also always enable provider-level prompt caching (Anthropic or OpenAI) first — it requires almost no code change and gives immediate cost savings on repeated system prompts."

Tip: flag the threshold risk with semantic caching — interviewers want to know you understand the failure mode.
