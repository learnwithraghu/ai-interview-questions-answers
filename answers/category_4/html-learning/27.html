Question 27  ·  AI Engineer Interview Bootcamp

Browser, Edge, or Server Inference?

Where should inference run — browser, edge, server — and why?

Choose the runtime based on latency, privacy, API key safety, and model size.

Question 27  ·  Three runtimes compared

Three runtimes, three trade-offs

SVR
Server inferenceMost control, centralised security, logging, rate limiting. Default for production LLM apps calling managed APIs.
EDG
Edge inferenceGeographically close to user, lower latency. Good for lightweight work (Cloudflare Workers AI, Vercel Edge). Strict compute limits.
BRW
Browser / on-device inferenceAbsolute privacy, no server needed. Requires WebGPU or Core ML. Limited to small quantised models.
API key safety: never put provider API keys in browser-side code — they will be extracted and abused. Cloudflare Workers AI: runs Llama/Mistral on edge GPUs. Zero latency overhead vs managed API for classification/embedding tasks. WebLLM: runs quantised LLMs in-browser via WebGPU. 100% private, works offline, limited to smaller models. Vercel AI SDK: routes streaming LLM calls from edge functions — API key stays server-side, low TTFT for users.

Question 27  ·  Decision framework

Four decision factors

1
API key securityIf the model call needs a provider key, it MUST go through server or edge (never browser).
2
Latency targetUnder 200ms? Consider edge. Under 50ms? On-device or edge-cached only.
3
Model sizeGPT-4 / Claude? Server only. Small classification model? Edge or browser viable.
4
Privacy requirementData must not leave the device? Browser / on-device only.
Typical product: grammar check → browser (tiny model). Autocomplete → edge. GPT-4 call → server. Apple Intelligence pattern: lightweight tasks on-device, complex requests to server. Tiered approach. Lambda@Edge / CloudFront: geo-distribute Lambda functions. Server inference with geographic distribution. Rule: start server-side, move to edge only when latency data justifies it and the model fits the constraints.

Question 27  ·  Say it

Say it

"I choose browser, edge, or server based on latency, privacy, cost, and control. Browser inference can be useful for small local models, but I'd never expose sensitive API keys there. Edge is good for low-latency lightweight work near the user. Server-side inference is my default for production LLM apps because it gives me centralised security, logging, routing, and rate limiting. The decision is always driven by data — I measure latency first, then evaluate whether edge or on-device inference actually helps."

Tip: the Apple Intelligence tiered model (on-device → server) is a strong real-world example to cite.
